# 創発的振る舞いに向けたロードマップ

「勇者パーティが魔王を倒す」「村人が引越しをする」「農家が村を出て勉強し、故郷に戻って先生になる」――こうした物語を**個別にプログラムせず**、汎用システムの組み合わせから**結果として**起こすことが目標。

---

## 1. 現時点で組み上がっているもの

### 1.1 コア基盤
- **Entity / Component**：全エンティティ（人・モンスター・建物・村・道・草原・雲）が同じ Entity 構造で、振る舞いはコンポーネントの有無で決まる。
- **Behavior + State**：FSM でひとつの「いま何をしているか」を保持。`DecisionState` がハブとして次の状態を選び直す。
- **割り込み（interrupt check）**：戦闘など、状態に関わらず割り込むべき条件を共通化。
- **SpatialQuery**：近傍検索の共通化。

### 1.2 身体・知覚・生存
- **Transform / Movement / Collider**
- **Health / DamageFlash / Loot**：死亡 → ドロップ。
- **Vitality**：体力的な総合値。
- **Nutrition**：空腹度。閾値（0.1 / 0.4 / 0.6）で「どこでも食べる」「家でなら食べる」「買いに行く」が分岐する。
- **Combat / AttackHitbox**：`shouldSeekCombat` と敵検出。攻撃的かどうかはフラグ次第。

### 1.3 持ち物・経済
- **Inventory / ItemCollector / Equipment**
- **Item / ItemInfo / Items データ**：`categories`（food / material / coin など）でカテゴリ分け。
- **Shop / ItemExchanger / BuyState / SellState**：村にある店との売買。
- **CarriedItemsFollower / FollowOwner**：持ち物の表現を Inventory から切り離して同期。

### 1.4 社会構造
- **Resident**：家（Inn）への居住・チェックイン／退去。
- **Inn / Party / PartyManager**：宿に同じ partyId が泊まれる。
- **Guild / QuestHolder / Quest 系 State**：依頼受注 → 移動 → 達成 → 報告。
- **Village / Building / Location / LocationLabel**：村と建物の物理的な配置。
- **Road / RoadPath / RoadRenderer**：村と村をつなぐ。

### 1.5 世界
- **GrassField / TreeField / StoneField / Mountain / CloudField / GroundTint / WorldBackdrop**
- **PerlinNoise / generateWorld**：地形生成。
- **MonsterSpawner / FieldSpawner**：場所に応じた湧き。

### 1.6 状態（State）
- `Idle / Wander / Home / Eat / Buy / Sell / Hunting / Combat / Revive / CollectItem / PartyMoveTo / SoloMoveTo / Dead`
- どれも汎用ロジックで、特定キャラ専用のものはない。

---

## 2. このゲームの良さ（設計の効いているところ）

1. **「兵士クラス」も「農民クラス」も存在しない**。差は持っているコンポーネントだけ。`combat` を持てば戦い、`questHolder` を持てば依頼を受け、`resident` を持てば家に帰る。
2. **DecisionState がひとつ**。プレイヤーも NPC も同じ判断ロジックを通る。AI を作るのではなく「条件をどう積むか」を考えるだけになる。
3. **データ（Inventory / Nutrition / Health）と表現（Renderer / Follower / FloatingText）が分離**しているので、表示を変えてもロジックは動き続ける。
4. **村・宿・店・ギルド・道が物理エンティティ**。位置と当たり判定で「いる／いない」が決まる。フラグで管理していないので、移動するだけで社会的な所属が変わる。
5. **欲求の閾値ベース**：空腹度・所持金などの数値が状態遷移を呼ぶ。シナリオではなく数値の流れがストーリーを作る。
6. **依頼が世界の動因になる**：ギルドに依頼が貼られ、攻撃的なエンティティが受注しに行き、対象の村に向かう。これが「冒険者が来た」という現象を勝手に発生させている。

つまり**「キャラを動かす」のではなく「世界に欲求とリソースを置く」**ことでゲームを作るスタイルが既に成立しはじめている。

## 3. 結果として起こしたいことと、まだ足りないもの

### 3.1 「勇者パーティが魔王を倒す」

現状でも近いことは起こせる（ギルド依頼 → 旅 → 戦闘 → 報告）が、**規模と固有性**が足りない。

足りないもの：
- **強さの段階（Level / Reputation）**：勝てない相手を避ける／勝てるようになると挑む、という判断材料。
- **脅威ランクと依頼ランクのマッチ**：ギルド依頼に難易度があり、`combat.power` と比べて受注可否が決まる。
- **Reputation / Fame**：依頼達成で蓄積し、難しい依頼を解禁する。これが「勇者になる」軌跡を作る。
- **強敵スポーン条件**：世界の脅威レベル（被害件数・経過時間）が閾値を超えたら魔王級が出現。倒されたら世界の脅威レベルが下がる。
- **パーティ編成欲求**：ソロでは勝てない相手 → 仲間募集 → 同じギルドにいる近い強さの個体と Party を組む。

### 3.2 「村人が引越しをする」

現状：`Resident.moveTo` は実装済み、`_departToVillage` で別村への移動も可能。だが**引越しを起こす欲求がない**。

足りないもの：
- **村ごとの状態指標**：人口、食料供給、治安（モンスター被害）、店舗数。
- **満足度コンポーネント（Satisfaction / Happiness）**：いまの村で得られる食事・安全・収入を集計。閾値以下で「移住したい」になる。
- **DecisionState の移住分岐**：満足度が低く、より良い村が観測できれば `_departToVillage` を呼ぶ。
- **村の観測手段**：噂システム、もしくは旅人がもたらす情報。常に全知だと味気ない。

### 3.3 「農家が村を出て勉強し、故郷に戻って先生になる」

これは一番遠い。必要なのは**役割が固定でなく、習得で変わる**こと。

足りないもの：
- **Skill / Knowledge コンポーネント**：farming, combat, literacy, teaching などのスキル値。経験で伸びる。
- **Profession は派生物**：「先生」というクラスはなく、`literacy` が高い住人が `School` 建物に居れば授業ステートに入る、という形にする。
- **学習場所（School / Library / Dojo）**：建物コンポーネント。滞在時間に応じてスキルが伸びる。
- **長期的目的（Aspiration / Goal）**：「都に出て学びたい」のような長期欲求。短期欲求（空腹）に上書きされつつ、満たされたときに優先される。
- **故郷への愛着（HomeOrigin）**：生まれた村を覚えており、Aspiration 達成後に帰還動機を生む。
- **時間スケール**：いまは秒単位。年・季節を導入し、長期目的が現実的な時間で進むようにする。
- **教えるという相互作用**：高スキル者が低スキル者と同じ建物にいるとスキル伝播。「先生になる」は結果。

## 5. 設計上の注意

これらを足すときも CLAUDE.md の原則を守る：

- 「先生クラス」「勇者クラス」を作らない。コンポーネントとパラメータの組み合わせから現れるようにする。
- スキルやレピュテーションは**データ**として持ち、判断は `DecisionState` の汎用分岐に追加する。
- 場所（School, MaouCastle）は他の建物と同じ `Building` + コンポーネントの組み合わせ。
- 「○○なら××する」を一行ずつ積んで、世界が回り出すのを観察する。

## 6. 追加で考えたい実装候補

### 6.1 ログ・関係性・記憶を一本化する：`Memory` コンポーネント

ログ（何をしたか）・関係性（誰と話したか／好感度）・記憶（出来事）は、形式上ほぼ同じ「**主体・相手・出来事・時刻**」の列。別々のコンポーネントにすると重複するので、**ひとつの `Memory` に統合**するのがシンプル。

```javascript
class Memory {
  constructor() {
    this.events = [];   // { time, type, target, data } の配列（リングバッファ）
    this.capacity = 50; // 古いものから捨てる
  }

  record(type, target = null, data = {}) {
    this.events.push({ time: this.entity.game.time, type, target, data });
    if (this.events.length > this.capacity) this.events.shift();
  }

  affinityWith(other) {
    // events を target===other で集計してスコア化
  }

  recentEvents(filter) { /* ... */ }
}
```

- **ログ表示** = `events` をフォーマットして出すだけ。
- **関係性／好感度** = `events` の集計関数。データを別に持たない。
- **記憶想起** = `recentEvents` でフィルタ。

イベント種別の例：`talked`, `helped`, `attacked`, `traded`, `saw_die`, `learned_from`, `arrived_at`, `quest_done`。

判断への接続：
- `DecisionState` で「最近 attacked された相手は避ける／反撃」「helped してくれた相手のパーティ募集に応じる」など。
- `Combat` で「affinity が低い相手なら `shouldSeekCombat` 扱い」にすれば、敵対関係がデータから生える。

### 6.2 強さは `Stats` コンポーネントに集約する

依頼ランクを別途持つより、**全エンティティが共通の `Stats` を持ち**、依頼の難易度はそこから計算するほうがシンプル。モンスターも人間も同じ尺度で比較できる。

```javascript
class Stats {
  constructor() {
    this.strength = 1;   // 攻撃
    this.endurance = 1;  // 耐久
    this.intellect = 1;  // 学習・交渉
    this.level = 1;      // 集約値（経験で上がる）
  }

  get power() {
    return this.strength * this.endurance;  // 概算戦闘力
  }
}
```

派生：
- **依頼難易度** = 対象モンスターの `stats.power`。受注可否は自分（またはパーティ合計）の `power` と比較。
- **Reputation** は `Stats` とは別物として残す（社会的評価）。`power` が高くても Reputation が低ければ大きな依頼は回ってこない、など。
- 経験値での成長は `Memory` のイベント履歴から導ける（`attacked` した回数で `strength` が上がる、など）。**ここでも記憶が経験値の源泉になる**。

### 6.3 会話・交流の最小実装：`Interaction` コンポーネント

「話した」を物理的に成立させる軽い仕組み。

- 同じ村／近距離に他エンティティがいて、どちらも忙しくない（`IdleState` か `HomeState`）なら一定確率で `talked` を `Memory` に書き込む。
- 内容は当面「天気」「依頼の噂」「最近の戦闘」など、自分の `Memory` から1件取り出して相手の `Memory` に書く。これで**情報伝播**になる（§4の噂システムの土台）。
- 好感度は `talked` の頻度から自然に上がる。

### 6.4 これらが揃うと起きること

- **「あの冒険者は強い」と村人が知っている** ← Memory の伝播 + Stats の参照
- **「裏切られた相手とは組まない」** ← Memory の attacked イベント
- **「先生に教わった生徒が、その先生を慕って戻る」** ← learned_from イベント + 好感度
- **「魔王討伐の英雄譚が村中に伝わる」** ← Memory(quest_done) が talked 経由で拡散
- **依頼の選別** ← 自分の Stats と依頼対象モンスターの Stats を比較するだけ

### 6.5 優先順（4節への追加）

| 優先 | 機能 | 効果 |
|---|---|---|
| ★★★ | **Memory コンポーネント** | ログ・関係性・記憶・噂・経験値の共通基盤。最初に入れる価値が高い。 |
| ★★★ | **Stats コンポーネント** | 依頼ランク・成長・戦闘判断をひとつで賄う。 |
| ★★ | **Interaction（会話イベント発生）** | Memory に書き込む現実的な手段。情報伝播の入口。 |
| ★★ | **Memory 集計ヘルパ（affinity, recent）** | 判断ロジックが一行で済むようにする。 |

### 6.6 設計上の注意

- `Memory` は**書き込みは多くの場所から、読み取りも多くの場所から**呼ばれる。なのでイベント種別の文字列だけは早めに決めて散らばらせない（`MemoryEvents.TALKED` のような定数）。
- 容量は必ず制限する（古いものから捨てる）。完全な記憶は作らない。
- `Stats` は読み取り専用に近い扱いにし、変更経路は経験値や成長メソッドに絞る。直接書き換えない。
- ログ表示用の UI はあとから乗せる。データさえ `Memory` に貯まっていれば、表示形式は差し替えられる（§設計原則4：データと表現の分離）。
